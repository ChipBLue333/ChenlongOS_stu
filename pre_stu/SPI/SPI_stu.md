## 1. 初步认识：SPI 到底是个啥？

**SPI** 的全称是 *Serial Peripheral Interface*，翻译过来叫**串行外设接口**。
简单来说，这是单片机（MCU）和各种外围芯片（如 Flash 存储器、OLED 显示屏、陀螺仪等）之间交流的“语言”。

它有三个最显著的标签：
1. **同步**：你们聊天得有个“节拍器”，大家按照统一的动作收发数据。这个节拍器就是时钟线。
2. **全双工**：就像面对面吵架，你在骂他的同时，他也能在骂你。发送和接收可以**同时**进行。
3. **主从结构**：必须有一个老大（Master，通常是单片机）和至少一个小弟（Slave，各种模块）。通信**永远由老大主动发起**，小弟不能主动找老大聊天，只能乖乖等老大点名。

## 2. 硬件连线：打通经脉的“四根线”

要让老大和小弟聊上天，我们需要在硬件上连四根线。我们来看看它们都是干嘛的：

| 简称 | 全称 | 大白话解释 | 信号传输方向 |
| :--- | :--- | :--- | :--- |
| **SCLK** | Serial Clock | **时钟节拍线**。老大负责踩点儿：“一、二、一、二”，大家按节拍动作。 | 主机 ➡️ 从机 |
| **MOSI** | Master Out Slave In | **老大的嘴，小弟的耳朵**。主机通过这根线发指令或者数据给从机。 | 主机 ➡️ 从机 |
| **MISO** | Master In Slave Out | **小弟的嘴，老大的耳朵**。从机通过这根线把主机要的数据送回来。 | 从机 ➡️ 主机 |
| **CS / NSS** | Chip Select | **片选线 / 点名线**。老大要跟哪个小弟说话，就把连着它的这根线**拉低**（没错，一般是低电平有效）。 | 主机 ➡️ 从机 |

> **💡 通俗比喻：**
> 老师 (Master) 想要提问学生 (Slave)。
> 老师先喊某个学生的名字（**CS 拉低**），接着老师开始敲桌子打节拍（**SCLK**）。
> 伴随着节拍，老师开始提问（**MOSI 说**），学生伴随着同一个节拍回答（**MISO 说**）。提问结束，老师让学生坐下（**CS 拉高**）。

## 3. 核心灵魂：移位寄存器（为什么读必须要写？）

这是无数初学者最容易卡壳的地方：**如果我只想从传感器读取一个数据，我为什么还要给它发送东西？**

这是因为，SPI 通信在单片机里面的物理本质，是**两个互通的“移位寄存器”**。
你可以想象成老大和小弟各拿了一个有 8 个格子的传送带，彼此首尾相连，形成一个大圆环。

1. 当 SCLK 时钟的第 `1` 个节拍响起，老大的第 1 个格子里的数据被推了出去，顺着 **MOSI 线** 滑到了小弟身上。
2. **与此同时**，小弟身上的第 1 个格子里的数据，也顺着 **MISO 线** 被推回给了老大。
3. 当 `8` 个节拍走完，老大的 8位数据 全部跑到了小弟那里，小弟原来的 8位数据 全部跑到了老大这里。
4. **他俩的数据完成了一次完美的“互换”！**

**所以得出一个重要结论：**
* 哪怕老大只是想查水表（单纯读数据），他也**必须**随便发点什么东西（比如发一个没有意义的 `0xFF`，业内叫 **Dummy Data / 哑数据**），目的是为了制造 SCLK 节拍。
* 没有节拍，传送带就转不起来，小弟的数据就永远送不回来。

## 4. 四种模式：CPOL 与 CPHA 到底在折腾啥？

不同的 SPI 芯片厂家，脾气不太一样。有的喜欢在灯亮着的时候收数据，有的喜欢在灯关着的时候收。这就诞生了 SPI 的 4 种工作模式。
你要是不把单片机的模式配得跟小弟的模式一模一样，两边“跨服聊天”，你读出来的数据就会歪掉，全变乱码。

模式由两个参数决定：
* **CPOL (Clock Polarity，时钟极性)**：老大不说话时，SCLK 时钟线是维持高电平（1），还是低电平（0）？
* **CPHA (Clock Phase，时钟相位)**：我们是在第 1 个边沿（比如刚刚从 0 变 1 时）抓取数据，还是在第 2 个边沿抓取数据？

排列组合一下，就有了下面的表格（**干货，一定要收藏**）：

| 模式 | CPOL | CPHA | 闲着没事时 SCLK 的状态 | 数据在哪个瞬间被“抓取”（采样） |
| :--- | :--- | :--- | :--- | :--- |
| **Mode 0** | 0 | 0 | 趴在地上（低电平） | 听到节拍的第一声（上升沿） |
| **Mode 1** | 0 | 1 | 趴在地上（低电平） | 听到节拍的第二声（下降沿） |
| **Mode 2** | 1 | 0 | 飘在天上（高电平） | 听到节拍的第一声（下降沿） |
| **Mode 3** | 1 | 1 | 飘在天上（高电平） | 听到节拍的第二声（上升沿） |

> **🛠 工程师实战经验：** 
> 拿到一个没用过的芯片（比如 W25Q128 Flash内存），第一件事就是翻它长长的英文数据手册（Datasheet），直接搜 "SPI Timing"，看它的时序图，或者说明文字里标注的是支持 Mode 0 还是 Mode 3。市面上 80% 以上的外设基本都支持 Mode 0。

## 5. 多从机连接：老大哥怎么管理多个小弟？

当单片机需要同时连接 OLED 屏幕、Flash 存储器和温湿度传感器时，SPI 怎么接线呢？

最常见的是**独立片选法**（一点多对法）：
1. 老大的 SCLK、MOSI、MISO 这三根线，扯拉面一样牵到所有小弟身上（大家共享这三条线）。
2. 但是老大要给每个小弟发一根**专用的 CS 线**（例如 CS1 给屏幕，CS2 给存储）。
3. 任何时刻，老大只能**拉低一根 CS 线**。比如拉低 CS2 时，只有存储器会竖起耳朵听讲并且用 MISO 说话。其他小弟会主动“闭嘴求生”，将自己的 MISO 引脚切换到**高阻态**（不输出高低电平，相当于剪断线路）。

## 6. 实战代码：如何用 C 语言驱动 SPI？

当你懂了原理，写代码就是水到渠成的事。几乎所有厂家的单片机，封装好的底层 SPI 收发函数，核心思想都是一样的：**发一个字节，必定收一个字节**。

### 第一步：写一个最基础的底层“死循环”互换函数
```c
// 无论后续怎么读写，底层永远是收发互换函数
uint8_t SPI_TransferByte(uint8_t data_to_send) {
    // 1. 把老大的数据塞进传送带
    SPI_DataRegister = data_to_send;  
    
    // 2. 坐等传送带转完（等待发送完成标志位）
    while( !SPI_CheckFlag(TX_EMPTY) ); 
    
    // 3. 坐等对方数据传过来（等待接收完成标志位）
    while( !SPI_CheckFlag(RX_READY) ); 
    
    // 4. 把对方传过来的数据拿出来返回
    return SPI_DataRegister;
}
```

### 第二步：写寄存器（老大向传感器下达指令）
```c
// 假设传感器要求：先发寄存器地址，再发数据
void Sensor_WriteReg(uint8_t reg_addr, uint8_t value) {
    CS_LOW();                     // 1. 张三你站起来（拉低片选）
    
    SPI_TransferByte(reg_addr);   // 2. 告诉他要写的地址。我们不关心这时候收到了什么。
    SPI_TransferByte(value);      // 3. 告诉他要写进去的值。
    
    CS_HIGH();                    // 4. 张三你坐下（拉高片选，通信结束）
}
```

### 第三步：读寄存器（老大读取传感器的值）
```c
// 假设传感器要求：先发寄存器地址(带个读标志)，然后等待传感器传回数据
uint8_t Sensor_ReadReg(uint8_t reg_addr) {
    uint8_t read_data = 0;
    
    CS_LOW();                     // 1. 张三站起来
    
    // 2. 将寄存器地址发给传感器，通常某些芯片要把最高位变成 1 代表“读操作”
    SPI_TransferByte(reg_addr | 0x80); 
    
    // 3. 重点来了！我们要读数据，就必须发一个没意义的哑数据 (比如0xFF) 去产生时钟！
    read_data = SPI_TransferByte(0xFF); 
    
    CS_HIGH();                    // 4. 张三坐下
    
    return read_data;             // 5. 返回拿到手的数据
}
```

## 7. 避坑指南：工程师吐血总结的常见错误

就算你把协议背熟了，在实际项目中还是可能遇到读出来全是 `0x00` 或者 `0xFF` 的情况。不用慌，排查下面这 4 个点能解决 95% 的问题：

1. **CS 线管理错乱**：
   有的小白在发送一帧较长的数据时，每发完一个字节就手贱把 CS 拉高一下，接着又马上拉低。
   **记住**：一次完整的行为（发指令 + 读数据），**中途千万别动 CS 线**，必须等所有事情做完了，最后再拉高！一旦拉高，外围传感器就认为本次对话强行结束并复位了。

2. **通信速率实在太高了**：
   觉得代码都没错，但是抓取数据缺斤少两？查查芯片手册！有些从机最大仅支持 `10 MHz`，如果你贪快把单片机时钟开到了 `30 MHz`，小弟根本反应不过来。建议新手调试从较低的频率（例如 `1 MHz` 或更低）开始。

3. **忘记拉低 CS 线（纯新手坑）**：
   初始化完外设后，激动地开始传输数据，发现全没用。原因是忘记第一步把 CS 脚拉低（写代码经常忘了 `CS_LOW()`）。

4. **高低位顺序（MSB/LSB First）发反了**：
   数据是一个 bit 一个 bit 在线上传的，究竟是最高位（Bit 7）先走，还是最低位（Bit 0）先走？
   SPI 协议绝大多数情况下默认是 **MSB First（高位在前）**，但偶尔会碰到奇葩芯片是 LSB First，导致收发完全鸡同鸭讲。如果死活不对，可以查手册确认一下这个选项。
